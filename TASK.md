# Тестовое задание для позиции React Junior (младший программист)

## Описание проекта

Необходимо реализовать небольшое веб-приложение **Candidate Viewer** — просмотр списка кандидатов и детальной информации по каждому. Приложение помогает HR-специалисту быстро просмотреть кандидатов, отфильтровать их по статусу соответствия вакансии и открыть подробную карточку.

Задание проверяет базовые навыки frontend-разработчика: вёрстку, работу с React, загрузку данных, маршрутизацию и аккуратность кода. Это **junior-уровень** — мы не ждём сложной архитектуры, оцениваем фундамент и чистоту решения.

> Если какой-то пункт окажется сложным — это нормально. Сделайте, что успеваете, и опишите в README, что осталось. Качество важнее количества.

---

## Технические требования

### Стек (обязательно)
- **React 18** + **TypeScript**
- **Vite** (сборка)
- **React Router** (маршрутизация между страницами)
- **CSS / SCSS** или **Tailwind CSS** (стилизация)

### Стек (по желанию, будет плюсом)
- Redux Toolkit / Zustand / Context API — если данные удобнее держать в общем сторе
- Jest + React Testing Library — если успеете написать тесты
- ESLint / Prettier

> **Важно:** менеджер состояния (Redux/Zustand) для этого задания **не обязателен** — достаточно `useState` / `useContext`. Тесты — тоже **бонус**, а не обязательное требование.

### Данные
Бэкенда нет — используйте мок-файл `mock/candidates.json` (12 кандидатов).

Загрузку данных оформите **асинхронно** (имитация запроса к серверу): функция в `services/`, которая возвращает `Promise` с задержкой через `setTimeout` (например, 500 мс). Это нужно, чтобы показать обработку состояний загрузки и ошибки.

## Задания

### Задание 1. Список кандидатов (главная страница)

**Требования:**
- Загрузить кандидатов из мока через асинхронный сервис (с имитацией задержки).
- Показать состояние **загрузки** (спиннер/скелетон/текст «Загрузка…») и состояние **ошибки**.
- Отобразить кандидатов карточками или таблицей. Для каждого: **ФИО, город, общий опыт, основной стек, вердикт** с цветовой индикацией.
- Если кандидатов нет — показать заглушку (empty state).

**Критерии оценки:**
- [ ] Данные загружаются асинхронно, есть loading и error состояния
- [ ] Список отображается корректно
- [ ] Вердикт выделен цветом (зелёный/оранжевый/красный)
- [ ] Обработан пустой результат

---

### Задание 2. Поиск и фильтрация

**Требования:**
- **Поиск по ФИО** — текстовое поле, список фильтруется по мере ввода.
- **Фильтр по вердикту** — переключатель/селект: `Все` / `ПОДХОДИТ` / `ЧАСТИЧНО` / `НЕ ПОДХОДИТ`.
- Поиск и фильтр должны работать **совместно**.
- Показать количество найденных кандидатов (например, «Найдено: 5»).

**Критерии оценки:**
- [ ] Поиск по имени работает корректно (регистронезависимо)
- [ ] Фильтр по вердикту работает
- [ ] Поиск и фильтр комбинируются
- [ ] Корректно обрабатывается «ничего не найдено»

> **Будет плюсом:** debounce для поля поиска (300 мс), вынесенный в кастомный хук `useDebounce`.

---

### Задание 3. Детальная страница кандидата

**Требования:**
- При клике на кандидата открывается страница `/candidate/:id` (через React Router).
- Отобразить полную информацию: контакты (email, телефон, telegram, город), **опыт работы** (таблица/список), образование, стек.
- Блок **«Критерии оценки»** — список с цветовой индикацией (ok / partial / no).
- Блок **«Вопросы для собеседования»** и блок **«Summary»**.
- Кнопка **«Назад к списку»**.
- Если кандидат с таким `id` не найден — показать сообщение «Кандидат не найден» (обработка 404).

**Критерии оценки:**
- [ ] Маршрутизация работает (список ↔ детальная)
- [ ] Отображается вся информация о кандидате
- [ ] Критерии показаны с правильными иконками/цветами
- [ ] Обработан случай несуществующего `id`

---

### Задание 4. Вёрстка и адаптивность

**Требования:**
- Семантическая разметка (`<header>`, `<main>`, `<section>`, `<nav>` и т.д.).
- **Адаптивная вёрстка**: приложение корректно выглядит на мобильном (~375px) и десктопе (~1280px).
- Аккуратные отступы, читаемая типографика, единый стиль.

**Критерии оценки:**
- [ ] Вёрстка адаптивна (мобильный + десктоп)
- [ ] Используется семантический HTML
- [ ] Нет «съезжающей» вёрстки и горизонтального скролла на мобильном
- [ ] Единообразный визуальный стиль

---

### Задание 5. Дополнительно (по желанию)

Любой из пунктов будет плюсом, но не обязателен:
- **Сортировка** списка по имени или опыту.
- **Бейдж статуса** кандидата (`new` / `reviewing` / `interviewed`) в списке.
- **Тесты** (Jest + React Testing Library): например, тест фильтрации или рендера карточки.
- Сохранение строки поиска/фильтра в **URL** (query params).
- Кастомные хуки (`useDebounce`, `useCandidates`).

## Что оценивается

### Обязательно (фундамент junior)
- **Вёрстка**: HTML5/CSS3 (или SCSS/Tailwind), адаптивность, семантика.
- **React-основы**: компоненты, props, `useState`/`useEffect`, списки и ключи, условный рендеринг.
- **TypeScript**: типы для данных кандидата и props (базовый уровень; старайтесь избегать `any`).
- **Работа с данными**: загрузка из сервиса, loading/error состояния.
- **Маршрутизация**: React Router, переходы между страницами.
- **Чистота кода**: понятные имена, разбиение на компоненты (не один большой файл), отсутствие закомментированного мусора.

### Будет плюсом
- Кастомные хуки, debounce
- Тесты
- Менеджер состояния (Redux/Zustand/Context) к месту
- Доступность (a11y): alt, aria-label, навигация с клавиатуры
- Аккуратная история коммитов

## Уровни оценки

| Критерий | Не прошёл | Junior | Junior+ (готов к собеседованию) |
|----------|-----------|--------|----------------------------------|
| **Вёрстка** | Не адаптивна, ломается | Адаптивна, семантика | Аккуратный UI, мелочи продуманы |
| **React** | Ошибки, всё в одном файле | Компоненты, хуки-основы | Разделение ответственности, кастомные хуки |
| **TypeScript** | `any` повсюду / нет типов | Базовые типы данных и props | Типы без `any`, аккуратно |
| **Данные** | Без обработки состояний | loading/error есть | Обработаны edge cases |
| **Код** | Дубли, грязно | Читаемо, разбито на компоненты | Чисто, переиспользуемые компоненты |
